# テスト駆動開発（TDD）

## 哲学

テストは**公開インターフェース経由で振る舞いを検証**する。実装の詳細ではなく、「何をするか」をテストする。
内部をリファクタリングしてもテストが壊れないなら、良いテスト。

## アンチパターン: 水平スライス

**全テストを先に書いてから全実装するのは禁止。** これは想像上の振る舞いをテストしてしまい、質の低いテストになる。

```
NG（水平）: test1, test2, test3 → impl1, impl2, impl3
OK（垂直）: test1→impl1 → test2→impl2 → test3→impl3
```

## ワークフロー

### 1. 計画
- [ ] ユーザーとインターフェースの設計を確認
- [ ] テストすべき振る舞いを優先度順にリストアップ
- [ ] ユーザーの承認を得る

**全てをテストする必要はない。** クリティカルパスと複雑なロジックに集中する。

### 2. トレーサーバレット（最初の1本）

```
RED:   最初の振る舞いのテストを書く → テスト失敗
GREEN: テストを通す最小限のコードを書く → テスト成功
```

### 3. 増分ループ

残りの振る舞いごとに:

```
RED:   次のテストを書く → 失敗
GREEN: 最小限のコードで通す → 成功
```

ルール:
- 1回に1テストだけ
- 現在のテストを通すコードだけ書く
- 将来のテストを先取りしない

### 4. リファクタリング

全テストが通ってから:
- [ ] 重複を抽出
- [ ] SOLID原則を自然に適用
- [ ] 各リファクタリング後にテスト実行

**REDの状態でリファクタリングしない。** 必ずGREENにしてから。

## サイクルごとのチェック

- [ ] テストは振る舞いを記述している（実装の詳細ではない）
- [ ] テストは公開インターフェースのみ使用
- [ ] 内部リファクタリングでテストが壊れない
- [ ] コードはこのテストに対して最小限
- [ ] 投機的な機能を追加していない
