# 商談準備ブリーフィング — HYPHEN仕様

## 原則
商談の成否は準備で決まる。30分かかっていた準備を5分で完了させる。

## 入力
ユーザーに以下を確認:
1. **クライアント名**: 誰との商談か
2. **商談の目的**: 何を決めたいか（新規提案/進捗報告/契約更新/課題ヒアリング）
3. **参考情報**（任意）: 前回議事録のパス、Supabaseのプロジェクト、メールのやり取り等

---

## ブリーフィング生成フロー

### Step 1: クライアント情報収集

以下のソースを自動チェック:
- **Obsidian**: `01_Projects/` 配下にクライアント関連ノートがないか検索
- **Git履歴**: 該当プロジェクトの最近のコミットログ
- **docs/**: 過去の仕様書・議事録・レビュー結果
- **tasks/**: todo.md の進捗状況

### Step 2: ブリーフィング文書の生成

以下の構造で出力:

```markdown
# 商談ブリーフィング: {クライアント名}
**日時**: {商談日時}
**目的**: {商談の目的}

---

## 1. クライアント概要
- 会社名・業種・規模
- 担当者名・役職
- これまでの経緯（箇条書き3行以内）

## 2. 前回からの進捗
- 前回の決定事項と現在の状況
- 完了したタスク
- 未完了のタスク（理由付き）

## 3. 今回の議題（提案）
- 議題1: {内容} → 期待するアウトカム
- 議題2: {内容} → 期待するアウトカム
- 議題3: {内容} → 期待するアウトカム

## 4. 想定される質問と回答
- Q: {想定質問1}
  A: {回答案}
- Q: {想定質問2}
  A: {回答案}
- Q: {想定質問3}
  A: {回答案}

## 5. 価格・見積もり（該当する場合）
- 提案パッケージ: {Basic / Standard / Premium}
- 月額/一括: {金額}
- 競合との比較ポイント

## 6. 注意事項
- クライアントの性格・コミュニケーションスタイル
- 前回の商談での気になった発言
- 絶対に触れてはいけない話題（あれば）

## 7. ネクストアクション（商談後に決めること）
- [ ] {アクション1}
- [ ] {アクション2}
```

### Step 3: 出力

ファイル保存先:
```
docs/briefings/{date}-{client}-briefing.md
```

**ユーザーに確認**: 「このブリーフィングで商談に臨めますか？修正点はありますか？」

## HYPHENの商談パターン別テンプレート

### パターンA: 新規提案
- 重点: クライアントの課題ヒアリング結果 + 提案の骨子 + 概算見積もり
- 「だから何？」: なぜ今やるべきか、やらないとどうなるか

### パターンB: 進捗報告（定例）
- 重点: 前回→今回の進捗 + デモ画面のURL + 次のマイルストーン
- 「だから何？」: 予定通り or 遅延の場合は対策

### パターンC: 契約更新・追加提案
- 重点: これまでの成果の数値化 + 次フェーズの提案 + ROI
- 「だから何？」: 継続/追加投資する価値の根拠

### パターンD: 課題ヒアリング（初回）
- 重点: 業界の一般的な課題リスト + 質問の準備
- 「だから何？」: ヒアリングで何を引き出したいか

## ルール
- 推測で埋めない。情報がない箇所は「要確認」と明記する
- クライアントの名前・社名を間違えない（最重要）
- 過去の商談で約束したことを必ず確認する
- 5分で読める量に収める（A4で2枚以内）
